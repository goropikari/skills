# dev-workflow (開発プロセス管理プラグイン)

`dev-workflow` は、Claude Code でのシステム開発において、人間のレビューを挟みながら厳格に段階的開発（6つのステップ）を進めるための開発プロセス管理プラグインです。

Git worktree環境に完全対応しており、状態管理や成果物がプロジェクト内の特定のディレクトリに隠蔽され、開発環境を汚さない設計になっています。

---

## 🛠️ 主な機能 (Features)

- **厳格な6ステップ開発の強制**: 各ステップが完了し、人間が次のステップへの進行を許可（`/dev next` コマンド）するまで、Claude Code は次のフェーズのファイル変更・作成を行いません。
- **要件定義フェーズでの `grill-me` 強制**: ステップ 1 では `grill-me` を使い、Claude が推奨回答を添えながら一問ずつ要件を深掘りし、合意内容から要件定義書を作成します。
- **Git worktree 対応**: 状態管理ファイル（`CURRENT_STEP.md`）および設計成果物は、作業ディレクトリ直下の `.dev-workflow/` 以下に格納されます。テスト実装以降は対象プロジェクトの適切な場所へ実装します。
- **自動 `.gitignore` 設定**: 初回実行時に自動的に `.gitignore` へ `.dev-workflow/` が追加され、開発途中の成果物がGitコミット履歴を汚すのを防ぎます。
- **JavaScript 不要の軽量設計**: インターフェース定義に `SKILL.md` を使用し、実際の挙動は Python 3 スクリプト (`workflow.py`) で動作するため、シンプルかつ拡張が容易です。
- **プラグインルート基準の実行**: `workflow.py` は Claude Code が提供する `${CLAUDE_PLUGIN_ROOT}/skills/dev/workflow.py` を基準パスとして実行します。

---

## 📂 6つの開発ステップ (6 Steps)

開発は以下のステップ順に進められます。ステップ 1〜3 は `.dev-workflow/` 配下の設計成果物を作成し、ステップ 4 以降は対象プロジェクトの既存構成に合う場所へ実装します。

| ステップ | ステップ名                 | 対象ファイル                             | 説明                                                                                              |
| :------- | :------------------------- | :--------------------------------------- | :------------------------------------------------------------------------------------------------ |
| **1**    | **実装の要件定義**         | `.dev-workflow/01_requirements.md`       | `grill-me` で一問ずつ要件を深掘りし、合意内容から要件定義書を作成する。                           |
| **2**    | **テスト分析・設計**       | `.dev-workflow/02_test_design.md`        | 要件定義書を入力として、テスト観点、テストケース一覧を自動的に設計する。                          |
| **3**    | **Gherkin定義**            | `.dev-workflow/03_features/spec.feature` | Gherkin (Given/When/Then) 形式で期待される振る舞いを定義する。                                    |
| **4**    | **テスト実装**             | 実プロジェクト内の適切なテストファイル   | 対象プロジェクトのテスト構成に合う場所へ自動テストを実装する（この段階ではテストは未パス）。      |
| **5**    | **関数定義・コメント作成** | 実プロジェクト内の適切なソースファイル   | 対象プロジェクトの設計に合う場所へ、関数のシグネチャ、docstring、スタブ（`pass`）のみを実装する。 |
| **6**    | **内部ロジック実装**       | 実プロジェクト内の適切なソースファイル   | 対象プロジェクトのソースファイルに関数内部のロジックを実装し、テストをすべてパスさせる。          |

## 🚀 使い方 (Usage)

本プラグインをインストールすると、Claude Code 上で以下のスラッシュコマンドが使用可能になります。

### 1. 現在のステータス確認 (`/dev`)

現在のステップ情報、ターゲット、制約事項を表示します。

```bash
/dev
```

**出力例:**

```text
=========================================
🛠️  DEV-WORKFLOW (Step 1/6)
=========================================
現在のステップ: 1. 実装の要件定義
ターゲット: .dev-workflow/01_requirements.md
現在のステータス: IN_PROGRESS

📝 説明:
grill-me で要件を一問ずつ深掘りし、合意内容を要件定義書として記述する。

👉 Claudeへの指示:
1. `grill-me` スキルを使用して、要件を一問ずつ深掘りしてください。
   - 各質問には Claude の推奨回答を添えてください。
   - コードベースを確認すれば答えられることは、ユーザーに聞かずに調査してください。
2. 目的、対象ユーザー、スコープ、非スコープ、主要ユースケース、入出力、制約、受け入れ条件、未決事項を合意してください。
3. 合意後、ターゲットファイル `.dev-workflow/01_requirements.md` に要件定義書を作成・更新してください。
4. 今何が終わっていて、次に何が残っているのかのサマリ（完了したこと、残タスク）を
   `.dev-workflow/CURRENT_STEP.md` の「進捗サマリ」セクションに随時記録・更新してください。
5. ステップの作業が完了したら、生成・更新した成果物のパスと内容または要約をユーザーに出力してください。
6. 成果物の出力後、AI自身でコマンド `python3 .../workflow.py review` を実行してレビュー待ち状態にしてください。

⚠️ 制約事項:
次へ進む指示があるまで、絶対にこれ以降のステップのファイルを生成・変更しないでください。
=========================================
```

### 2. 次のステップへ進む (`/dev next`)

現在のステップの開発および成果物の確認を終え、次のステップへ移行します。

> [!NOTE]
> ステータスに応じて `/dev next` 実行時の振る舞いが変わります：
>
> - **`IN_PROGRESS` (作業中) の場合**: ステップは移行せず、Claude に対し、`.dev-workflow/CURRENT_STEP.md` の「進捗サマリ」セクションに進捗（完了項目・残タスク）を記録しながら作業を進めるよう指示します。作業完了後は AI が `/dev review` を実行してレビュー待ち状態にします。
> - **`REVIEW_PENDING` (レビュー待ち) の場合**: ステップは移行せず、人間（ユーザー）のレビュー確認を待つよう指示します。人間から「OK」「進めて」などの承認が出た段階で AI が `/dev approve` を実行して承認済み状態にします。
> - **`REVIEWED` (承認済み) の場合**: 正常に次のステップへ移行し、ステータスは新しいステップの `IN_PROGRESS` に戻ります。

各ステップの作業が完了したら、AI は生成・更新した成果物のパスと内容または要約をユーザーに出力し、その後 `workflow.py review` を自動実行して `REVIEW_PENDING` に移行します。

ステップ 2「テスト分析・設計」は、ステップ 1 の成果物 `.dev-workflow/01_requirements.md` から自動生成します。AI は追加質問をせず、要件定義書から判断できない内容を未決事項またはテストリスクとして `.dev-workflow/02_test_design.md` に明記します。

```bash
/dev next
```

### 3. レビュー待ち状態にする (`/dev review`)

現在のステップの開発および動作検証が完了したことを表すために、ステータスを `REVIEW_PENDING`（レビュー待ち）に更新します（通常は開発作業を終えた AI が実行します）。

```bash
/dev review
```

**出力例:**

```text
=========================================
✉️  DEV-WORKFLOW STATUS UPDATED TO REVIEW_PENDING
=========================================
現在のステップ: 2. Gherkin定義
ステータスを REVIEW_PENDING に更新しました（レビュー待ち状態）。
人間（ユーザー）によるレビュー・確認を待機します。
承認が得られたら、AI自身で /dev approve を実行してください。
=========================================
```

### 4. ステータスを承認済みに更新する (`/dev approve`)

人間のレビュー確認が完了したことを表すために、ステータスを `REVIEWED`（承認済み）に更新します（通常は人間からの指示を受けた AI が実行します）。

```bash
/dev approve
```

**出力例:**

```text
=========================================
✅  DEV-WORKFLOW STATUS UPDATED TO REVIEWED
=========================================
現在のステップ: 2. Gherkin定義
ステータスを REVIEWED に更新しました。
次のステップに進むには、/dev next を実行してください。
=========================================
```

## 📁 ディレクトリ構造 (Directory Structure)

```text
dev-workflow/
├── package.json         # Node.jsパッケージマニフェスト（スコープ名定義）
├── README.md            # 本ドキュメント
├── .claude-plugin/
│   └── plugin.json      # Claude Code プラグインメタデータ
└── skills/
    └── dev/
        ├── SKILL.md     # Claude Code コマンドインターフェース定義
        └── workflow.py  # プロセス制御ロジック (Python 3)
```

---

## ⚠️ 制約・注意事項 (Constraints)

- `.dev-workflow/CURRENT_STEP.md` に現在の進捗状態が保存されます。このファイルを直接書き換えると状態が壊れる可能性があります。
- AI (Claude) は、現在のステップの対象範囲以外のファイルを編集してはいけません。例えば、ステップ 1〜3 の時点で実装コードやテストコードを作成することは禁止されています。
- ステップ 1 では、要件定義書を直接書き始める前に `grill-me` による質問と合意形成を行います。
